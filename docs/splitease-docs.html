1Project Overview & Goals

SplitEase is a full-stack web application inspired by Splitwise, built to solve the common problem of tracking shared expenses among friends, roommates, and travel groups. The goal was to create a free, modern, self-hostable alternative with a beautiful UI.

Core Problem Solved: When groups of people share expenses (trips, shared housing, dinners), tracking who paid what and who owes whom becomes complex and awkward. SplitEase automates the calculation and simplifies settlement.

Key Goals

💸 Expense Tracking

Log expenses in groups with payer tracking, categories, and date. Support equal splits across all members.

⚖️ Smart Balances

Automatically calculate net balances between all group members, simplifying complex webs of debt into minimum transactions.

👥 Group Management

Create named groups with custom emoji/colour, invite members by email, assign admin roles.

✅ Settle Up

Record settlements that reduce balances. Full history of all settlements per group.

4System Architecture

4.1 High-Level Architecture

SplitEase is a monolithic full-stack app built on Next.js 16. There is no separate frontend and backend server — everything runs in a single Next.js process. The App Router architecture uses React Server Components for data fetching and Client Components for interactivity.

Architecture Pattern: Server-Side Rendering (SSR) + Client Components + Route Handlers (API). Server Components fetch data directly from the database (no API call overhead). Client Components use fetch() to hit Route Handlers for mutations.

5Database Design

5.1 Overview

The database uses SQLite accessed through the libSQL driver (compatible with Turso for production). Prisma 7 generates a fully type-safe client from the schema. The schema has 10 models covering users, authentication, groups, expenses, splits, settlements, and friendships.

Why SQLite? Zero configuration for local development — no server to install. The libSQL adapter (Turso) makes it trivially upgradeable to a distributed cloud database when going to production.

6Authentication System

How It Works

Authentication uses NextAuth v4 with two providers: Credentials (email + password) and Google OAuth. Sessions are stored as signed JWT tokens in a cookie — no server-side session storage required.

Credentials Provider Flow

1. User submits email + password to POST /api/auth/register
   → bcrypt.hash(password, 12) — 12 rounds is OWASP recommended
   → prisma.user.create({ email, hashedPassword })
   → Returns user object (no password exposed)

2. User signs in via signIn("credentials", { email, password })
   → NextAuth calls authorize() callback
   → prisma.user.findUnique({ where: { email } })
   → bcrypt.compare(password, user.password)
   → Returns user object → JWT issued with { id, email, name, image }

3. JWT stored in httpOnly cookie (NEXTAUTH_SECRET signs it)
   → Every request: proxy.ts calls getToken({ req }) to validate
   → Server components: getServerSession(authOptions) returns user

Session Type Augmentation

NextAuth's default Session type doesn't include user.id. This is fixed in src/types/next-auth.d.ts:

// src/types/next-auth.d.ts
declare module "next-auth" {
  interface Session {
    user: { id: string } & DefaultSession["user"]
  }
}

// src/lib/auth.ts — jwt callback adds id to token
async jwt({ token, user }) {
  if (user) token.id = user.id
  return token
},
// session callback puts it on the session object
async session({ session, token }) {
  if (token) session.user.id = token.id
  return session
}

Route Protection — proxy.ts

Next.js 16 renamed middleware.ts to proxy.ts. The proxy runs on every request before any page renders:

// src/proxy.ts
export async function proxy(req: NextRequest) {
  const token = await getToken({ req, secret: process.env.NEXTAUTH_SECRET })

  // Redirect logged-in users away from auth pages
  if (token && ["/", "/signin", "/signup"].includes(pathname))
    return NextResponse.redirect(new URL("/dashboard", req.url))

  // Redirect unauthenticated users to signin
  if (!token && !isPublicPath(pathname))
    return NextResponse.redirect(new URL("/signin", req.url))
}
Security Note: Every API Route Handler also independently calls getServerSession(authOptions) and returns 401 if no session. The proxy is a UX redirect layer — API security is never proxy-only.

8Business Logic — Balance Calculation

The balance calculation is the core algorithm of the app. It lives in GET /api/groups/[id]/balances.

Step 1 — Build Raw Debt Map

For every ExpenseSplit where the split owner ≠ payer and paid = false:

balances[split.userId][expense.paidById] += split.amount
// e.g. Alice paid $60 dinner, 3-way split:
// Bob owes Alice $20 → balances["bob"]["alice"] += 20
// Carol owes Alice $20 → balances["carol"]["alice"] += 20

Step 2 — Apply Settlements

For every Settlement record, subtract from the debtor's balance:

balances[settlement.fromUserId][settlement.toUserId] =
  Math.max(0, existing - settlement.amount)

Step 3 — Simplify Mutual Debts

If A owes B $30 AND B owes A $20, the net result is A owes B $10 only:

for each pair (A, B):
  net = balances[A][B] - balances[B][A]
  if net > 0.01:   emit { fromUser: A, toUser: B, amount: net }
  if net < -0.01:  emit { fromUser: B, toUser: A, amount: -net }
Result: The minimum number of transactions needed to settle all debts. A group of 10 people with 50 expenses might need only 5-8 payments to fully clear all balances.

Dashboard Balance (Across All Groups)

The dashboard calculates totalOwed (others owe me) and totalOwe (I owe others) by scanning all expenses in all groups the user belongs to:

if expense.paidById === myId:
  for each split where split.userId !== myId and !split.paid:
    totalOwed += split.amount

if expense.paidById !== myId:
  mySplit = expense.splits.find(s => s.userId === myId)
  if mySplit and !mySplit.paid:
    totalOwe += mySplit.amount

// Subtract confirmed settlements
netBalance = totalOwed - totalOwe

The Sidebar component is a Client Component (uses usePathname() for active state and useSession() for the user card). It renders on all (dashboard) routes. Features:

  • SplitEase logo and branding
  • New Group quick-action button
  • Three nav items: Dashboard, Groups, Friends (with active state highlight)
  • User avatar + name + email at bottom
  • Sign out button (calls signOut({ callbackUrl: "/" }))

Mobile Navigation

The MobileNav component renders as a fixed bottom tab bar on screens under md breakpoint (768px). Shows icon + label for Dashboard, Groups, Friends, and New group.

Navigation Guard

Two layers of protection:

  1. proxy.ts — Edge-level check using getToken(). Redirects unauthenticated requests to /signin before the page even renders.
  2. Dashboard layout — Server-side getServerSession() check with redirect("/signin") as a fallback in case proxy is bypassed.

15Deployment Guide (Vercel)

Recommended: Deploy to Vercel (built by the Next.js team) + Turso (managed libSQL database). Both have generous free tiers.

Step 1 — Set up Turso database

# Install Turso CLI
brew install tursodatabase/tap/turso

# Sign in / sign up
turso auth login

# Create a database
turso db create splitease-prod

# Get the connection URL and auth token
turso db show splitease-prod --url
turso db tokens create splitease-prod

Step 2 — Update schema for Turso

Turso uses libSQL which is SQLite-compatible. Update .env (production):

DATABASE_URL=libsql://splitease-prod-[your-org].turso.io
TURSO_AUTH_TOKEN=your-token-here

Update src/lib/prisma.ts to use the auth token for production:

const adapter = new PrismaLibSql({
  url: process.env.DATABASE_URL!,
  authToken: process.env.TURSO_AUTH_TOKEN,
})

Step 3 — Deploy to Vercel

# Install Vercel CLI
npm install -g vercel

# Deploy (follow prompts)
vercel

# Set production environment variables
vercel env add DATABASE_URL
vercel env add TURSO_AUTH_TOKEN
vercel env add NEXTAUTH_SECRET
vercel env add NEXTAUTH_URL         # Set to https://your-app.vercel.app
vercel env add GOOGLE_CLIENT_ID     # If using Google OAuth
vercel env add GOOGLE_CLIENT_SECRET

# Deploy to production
vercel --prod

Step 4 — Run migrations on production DB

# Point CLI at production DB temporarily
DATABASE_URL=libsql://... npx prisma db push

Alternative: Self-hosted with Docker

# Build Docker image
docker build -t splitease .

# Run with SQLite (volume-mounted)
docker run -p 3000:3000 \
  -e NEXTAUTH_SECRET=your-secret \
  -e DATABASE_URL=file:/data/prod.db \
  -v splitease-data:/data \
  splitease
